# 채팅 기반 TFT 유사도 개선: 현재 문제점과 작업 계획

## 배경
RCTK의 목적은 단일 정답 템플릿 복제가 아니라, 사용자가 제공한 이미지/의도를 바탕으로 **채팅으로 요소를 수정**하면서 목표 화면과 유사도를 점진적으로 끌어올리는 것이다.

따라서 핵심은 아래 3축이다.
1. 채팅 수정이 실제 데이터 모델에 반영되는가
2. 반영된 모델이 TFT로 정확히 export되는가
3. 결과물이 목표 이미지와 구조/동작 모두에서 유사한가

---

## 현재 문제점 (현 시점 기준)

### 1) 타입 추가는 되었지만, 실제 사용 흐름 점검이 부족
- `rectangle`, `button-nav`, `rtc` 타입 분기와 라벨이 추가되었으나,
- 채팅→에디터→export 전 구간에서 각 필드가 확실히 전달되는지 검증 자동화가 부족하다.

### 2) `button-nav`의 핵심 동작값(`switch`) 관리가 초기 단계
- `switchTo` 필드를 통해 exporter로 연결을 시작했지만,
- 채팅 수정 시 이 값을 일관되게 생성/보정하는 규칙이 아직 약하다.

### 3) `rtc`/`rectangle`는 타입 존재 여부보다 속성 스키마 정합이 중요
- VisualTFT는 타입명만 맞아도 충분하지 않고 속성 키/값 호환이 중요하다.
- 현재는 원본 스키마 정합성을 더 체계적으로 맞출 필요가 있다.

### 4) 비교/검증 루프가 수동 중심
- 현재는 시각 비교와 수동 확인 비중이 커서 반복 개선 속도가 느리다.
- 목표 이미지 대비 차이를 수치로 보여주는 diff 루프가 필요하다.

---

## 앞으로 해야 할 일 (우선순위 순)

## P0 — 채팅 수정 루프 안정화 (필수)
1. **필드 전달 보장**
   - `button-nav.switchTo`가 채팅 수정 → 상태 저장 → exporter까지 유지되는지 확인.
2. **최소 시나리오 회귀 체크**
   - rectangle 1개, button-nav 1개, rtc 1개를 가진 샘플에서 매번 export 검사.
3. **fallback 차단**
   - 위 3타입이 `type="text"`로 떨어지지 않도록 가드.

## P1 — 스키마 정합 강화
4. **rectangle/button/rtc 속성 키 표준화**
   - 원본 monitor 계열에서 쓰는 속성 키 중심으로 정렬.
5. **button-nav 동작 의미 강화**
   - `switchTo` 기본값 정책 및 빈값 경고(채팅/패널) 추가.
6. **rtc 표시 정책 고정**
   - format/format_string/font/timer 기본값을 프로젝트 표준으로 고정.

## P2 — 검증 자동화
7. **TFT diff 스크립트 운영화**
   - 타입 분포, 필수 속성 누락, 주요 속성 차이를 출력.
8. **채팅 기반 수정 시 검증 루틴 연결**
   - 주요 변경 뒤 diff 결과를 즉시 확인 가능한 워크플로우 문서화.

## P3 — 문서/운영
9. **"채팅으로 유사도 올리기" 플레이북 문서화**
   - 사용자가 어떤 순서로 수정 지시를 주면 효과적인지 단계화.
10. **릴리즈 전 체크리스트**
   - switch/rtc/rectangle 검증 통과 전 배포 금지 규칙 추가.

---

## 권장 실행 순서 (실무형)
1) P0-1~3 (데이터 전달/회귀/가드)
2) P1-4~6 (스키마/동작/표시 정책)
3) P2-7~8 (자동 검증)
4) P3-9~10 (운영 문서)

---

## 완료 기준 (Definition of Done)
- 채팅으로 `button-nav`의 이동 대상 수정 시 export TFT의 `switch`가 즉시 반영된다.
- `rectangle`, `button-nav`, `rtc`가 더 이상 fallback text로 출력되지 않는다.
- 표준 샘플 기준 diff 보고서에서 필수 속성 누락이 0에 수렴한다.
- 사용자가 "이미지 기반 수정"을 2~3회 반복하면 체감 유사도가 명확히 상승한다.
